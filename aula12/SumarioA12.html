Controlo de versões de software

Todo o desenvolvimento de software deve ser feito numa infraestrutura de salvaguarda de versões para recuperação de dados ou análise histórica. Há várias infraestruturas para o fazer e uma das mais usadas é o "git".

Assim, deve-se instalar e configurar o uso do git, e também criar um repositório numa outra máquina (de preferência on-line) para guardar o código e outros ficheiros relevantes. Uma plataforma muito utilizada neste âmbito é o github que, para além de disponibilizar o controlo de versões online baseado em git, oferece também várias ferramentas de gestão de projeto.

É usual manter sob controlo (e armazenamento remoto) apenas os ficheiros de código fontre (*.c, *.h, etc.), de documentação, de configuração e similares. Não se deve manter sob controlo de versões o código compilado e outros binários.

Criação do repositório e sincronização da cópia local

Instalar o comando git (só se faz uma vez e fica permanente)

sudo apt install git

Configurar o git para primeiro uso incluindo alguns elementos identificativos do autor.

git init
git config --global user.email you@email.com
git config --global user.name "Your Name"

Posteriormente deve criar um repositório remoto, fazendo o seguinte:

  • Criar uma conta on-line no GitHub.

  • Por exemplo "nomealuno_ua"

  • Nessa conta criar um repositório para as aulas de PARI.

  • (por exemplo "nomealuno_pari")

  • Criar um ficheiro README.md com o github (é o formato usual, mas podem ser outros formatos como README.adoc, etc.)

Para sincronizar o repositório remoto com o local deve fazer

cd <caminho onde quer o repositório>
git clone <url do repositório>

Depois deve ter uma pasta com o nome do repositório. Lá dentro deve criar a árvore de diretórios e ficheiro CMakeLists.txt:

src
doc
CMakeLists.txt

Usar por exemplo os ficheiros do último exercício realizado na aula anterior e acrescentar e ajustar os ficheiros para gerar a documentação.

Ex. 1b)

Atualização do repositório remoto com a cópia local

O comando

git status

permite ver quais os ficheiros ou diretorias que estão sob controlo de versões. Para especificar os ficheiros e diretórios a incluir do controlo de versões.

git add README.md
git add CMakeLists.txt src doc

Sempre que se criar um ficheiro novo numa pasta sob controlo de versões, é necessário adicioná-la com git add.

Fazer o primeiro "commit". Isto significa criar um registo local apenas, nada está ainda on-line.

git commit -m "first commit"

Depois, para enviar as alterações para o repositório remoto

Aplicações com OpenCV e GTK

Existe uma situação de incompatibilidade do OpenCV 2.x com GTK+3 que se deve ao facto de uma parte do openCV que foi compilada usando dependências do GTK+2. São exemplo as funções presentes na secção highgui do opencv que assegura a visualização de imagens em janelas, ou a aquisição de imagem ou carregamento de imagens do disco. Isso não acontece unicamente com o GTK e sucede com outras bibliotecas como o Qt. Essa componente highgui que é fonte dessas incompatibilidades atuais, tem um conjunto de funcionalidades listadas no link seguinte, e que portanto não poderão ser usadas se o módulo highgui for excluido da compilação: http://docs.opencv.org/2.4/modules/highgui/doc/highgui.html

Para resolver este problema de incompatibilidade há estas soluções principais:

  1. Desenvolver aplicações de GTK+ na versão GTK+2.X;

  2. Recompilar o OpenCV com suporte para GTK+3 e desenvolver aplicações em GTK+3;

  3. Não usar funcionalidades de GUI (módulo highgui) na aplicação e aí pode-se desenvolver aplicações em qualquer versão de GTK.

Em qualquer dos casos, é perfeitamente normal e recomendável usar as novas facilidades do GTK+3 que na maioria já estavam previstas nas versões finais do GTK+2, como o Cairo.

Nestas circunstâncias, também se recomendaria usar o glade-gtk2 para criar as interfaces gráficas sem incompatibilidades (uma vez que o glade exige GTK+3).

Existe grande similaridade na maioria dos elementos gráficos entre as versões para GTK+2 e GTK+3, havendo novos equivalentes ou alternativas, como por exemplo os seguintes:

                GTK+2                  |               GTK+3
widgets Horizontal/Vertical box        |  widget box
Widget table                           |  widget grid
expose event                           |  draw event

O problema é que a versão 16.04 do Ubuntu não inclui o glade-gtk2 e, portanto, não é possível desenvolver aplicações em glade para GTK+2 nesta versão da mesma forma que se fazia em versões mais antigas do ubuntu. No entanto, podem-se usar ficheiros já desenvolvidos, e podem-se criar aplicações escrevendo manualmente o código glade com um editor vulgar de texto ASCII. Esta será uma solução transitória enquanto os repositórios do Ubuntu não tiverem uma versão mais recente do openCV onde não há estes problemas.

Nota: Há sempre a solução de instalar uma versão mais antiga do ubuntu (e.g. 14.04), por exemplo em máquina virtual, e aí já se poder executar o glade-gtk2. Não será obrigatório nem estritamente necessário, mas é uma solução de recurso.

É recomendado instalar documentação adicional do GTK2 e GDK2 o que se pode fazer com:

sudo apt-get install libgtk2.0-doc libgdk-pixbuf2.0-doc

Generalidades e recomendações para desenvolver este projeto

Na pasta src ter os seguintes ficheiros:

  • main.c - só tem o programa principal

  • callbacks.c - contem as callbacks dos widgets, e handlers de sinais, timers e timeouts.

  • myf.c - contém todas as restantes funções auxiliares.

  • myf.h - header file do projeto (variáveis globais, definições de tipos, inclusão de outras headers, etc.).

  • prototypes.h - header file com os protótipos das funções (gerado com genproto.sh ou similar).

  • mydr2.glade - Ficheiro glade com a interface gráfica a usar.

Para facilitar a identificação das diversas funções as seguintes normas vão ser adotadas:

Nomes das callbacks de widgets e apenas essas: (seguindo a norma do próprio GTK)

on_drawingarea1_expose_event()
on_button1_clicked()
etc.

Nomes das funções auxiliares: (inclui as de processamento e handlers/callbacks que não são de widgets)

pari_UpdateImageAreas()
pari_IntersectCTRL_C()
pari_PerformImageAcquisition()
etc.

O prefixo pari_ nestas funções ajuda a identificar rapidamente o seu enquadramento e distingue-as de outras que possam existir noutras bibliotecas.

Para facilitar o desenvolvimento, sugere-se que o ficheiro header do projeto (myf.h ou similar) faça todos os #includes necessários. Para o GTK e opencv devem ser pelo menos os seguintes:

excerto exemplo de myf.h
#ifndef _MYF_H
#define _MYF_H

//...

#include <gtk/gtk.h>
#include <cv.h>
#include <highgui.h>

//...

/* prototypes should be within the last lines of header */
#include "prototypes.h"

#endif /* _MYF_H */

De igual forma se devem ter as variáveis globais devidamente declaradas, em especial as ligadas à interface gráfica. Alguns exemplos de seguida:

excerto de myf.h para as variáveis globais
//...

#if defined (_MAIN_C_)
        GtkBuilder *builderG;
        IplImage *dst_imageG , *src_imageG;
        CvCapture *captureG;
#else
        extern GtkBuilder *builderG;
        extern IplImage *dst_imageG , *src_imageG;
        extern CvCapture *captureG;
#endif

//...

Ex. 2a)

Criar um programa em GTK+ (em v2 porque em v3 terá conflitos com o openCV2) que, usando o openCV, faça a visualização numa janela de uma imagem em tempo real da câmara.

As funções necessárias à aquisição de imagem devem ser criadas no ficheiro myf.c e baseiam-se em exemplos criados em aulas anteriores. São as seguintes:

CvCapture *pari_StartImageAcquisition() ;
void pari_PerformImageAcquisition(CvCapture *capture) ;

A função pari_StartImageAcquisition() devolve o "device" de captura e cria a imagem onde será colocada a leitura da câmara, e pari_PerformImageAcquisition() faz a aquisição propriamente dita. À semelhança do que foi feito anteriormente, o código sugerido para estas funções é:

CvCapture *pari_StartImageAcquisition()
{
        CvCapture *capture = cvCaptureFromCAM(0);              //Capture using 1st camera: i.e., no. 0
        IplImage *frame=cvQueryFrame(capture);                 //Update frame. Pointer does not change.
        src_imageG = cvCreateImage(cvGetSize(frame), IPL_DEPTH_8U, 3);//create image for color (8 bits, 3 channels)
        dst_imageG = cvClone(src_imageG);                       //create clone of source image
        return capture;
}

void pari_PerformImageAcquisition(CvCapture *capt)
{
        static IplImage *frame;
        frame=cvQueryFrame(capt);               //Update frame pointer.
        cvCopy(frame, src_imageG, NULL);        //copy frame as it is (preserve color)
        cvCopy(src_imageG, dst_imageG, NULL);   //copy src into dst and ready to process (admitt similar image structure)
}

Estas funções pressupõem a existência de variáveis globais para conter as imagens adquirida e processada: IplImage *src_imageG, *dst_imageG;. Como é usual, devem ficar declaradas na header file do projeto. As funções devem ser criadas no ficheiro de funções auxiliares, por exemplo myf.c, myfuncs.c ou aux.c ou similar, mas que não seja o ficheiro das callbacks dos widgets.

Se não estiver disponível uma webcam, a função pari_StartImageAcquisition() não deverá fazer nada e deve simplesmente retornar. Nesse caso, a função pari_PerformImageAcquisition() deverá também ser alterada para fazer a leitura de imagens de disco. Dependendo do que se pretende fazer, pode ser necessário ajustar a função; por exemplo, se só houver uma imagem em disco carrega-se a primeira vez e nas chamadas seguintes a função retorna sem recarregar a imagem, mas se por outro lado houver um conjunto de imagens que se queiram carregar em sequência será preciso acautelar a libertação da memória das imagens anteriores para carregar a nova.

Ex. 2b)

Criar no ficheiro callbacks.c uma função "handler" que será chamada por um timer (timeout) para fazer o refresh de uma Drawing Area apropriada e atualizar a imagem da câmara e a imagem processada.

A função handler que será executada automaticamente no fim de cada timeout
gboolean pari_UpdateImageAreas(gpointer data)
{
        //generate an expose event (draw event) on drawingarea1
        GtkWidget *da1 = GTK_WIDGET(gtk_builder_get_object(builderG, "drawingarea1"));
        gtk_widget_queue_draw(da1);
        return TRUE;
}

Deve-se também ativar o timer com g_timeout_add() usando um tempo entre 30 e 100 milisegunos. Este timer deverá ser criado no programa principal antes do gtk_main().

Algum código relevante no main() incluirá por exemplo o seguinte:

int main(int argc, char *argv[])
{
        gtk_init(&argc, &argv);

        //...

        g_timeout_add(30, (GSourceFunc) pari_UpdateImageAreas, (gpointer) NULL);
        captureG=pari_StartImageAcquisition();

        /* start the event loop */
        gtk_main();

        cvReleaseCapture(&captureG);             //Release capture device.
        cvReleaseImage(&dst_imageG);             //Release image (free pointer when no longer used)
        cvReleaseImage(&src_imageG);             //Release image (free pointer when no longer used).
        return 0;
}

Pressupõe-se que o identificador do device de captura CvCapture *captureG; também é variável global.

Ex. 2c)

O objetivo seria criar uma interface gráfica que permita inserir uma imagem de openCV numa DrawingArea. Mas como o glade-gtk2 já não existe para ubuntu 16.04, propõe-se usar uma interface já previamente criada e que se fornece on-line para este exercício (mydr2.glade). O ficheiro pode-se abrir com o glade mais recente, mas são detetadas widgets e propriedades obsoletas. Porém, dá para visualizar os widgets e a maioria das suas propriedades. Só não dá é para alterar nem guardar em cima do ficheiro original!

Esta interface tem duas drawing areas e alguns botões e outros widgets para poder definir operações. Uma vez que se pretende atualizar automaticamente o refresh para refletir as imagens mais recentes da câmara, não será necessário criar callbacks associadas a esses widgets das checkbox! Ou seja, a chegada de uma nova imagem é que deve despoletar o redesenho das janelas (drawing areas), embora se pudesse também forçar o "refresh" dessas drawing areas quando as checkbox ou outros botões fossem alterados mas, como se referiu, não será necessário porque o "timer" despoleta a captura de nova imagem e o consequente evento de "redesenho" da drawing area.

Para além das duas drawing areas há check boxes (um tipo especial de botão) para operações de tratamento de imagem (e.g. conversão para cinzento, binarização, erosão, inversão, Canny). Há também barras que definem parâmetros para algumas operações (e.g. limite de binarização, número de erosões, etc.). Essas barras são widgets do tipo scale e precisam de ter associado um widget do tipo adjustment onde se configuram os limites associados à barra em si. Os nomes dos widgets serão usados no programa.

Exercício 3 - Criar as funções de conversão e representação de imagens

Ex. 3a)

Criar em myf.c a função para transformar uma IplImage num pixbuf para o poder colocar num widget do tipo drawing area. O Gdk-Pixbuf é um toolkit para carregamento de imagens e manipulação de buffers de pixels e é usado pelo GTK+2 e pelo GtK+3 para carregar e manipular imagens.

// Exemplo de código de uma função que converte
// uma IplImage num pixbuf para inserir numa interface GTK
// e ajusta-a com certas dimensões passadas como argumentos

GdkPixbuf *pari_ConvertOpenCv2Gtk(IplImage * image, int dst_w, int dst_h)
{
    IplImage *gtkMask=image;
    GdkPixbuf *pix, *rpix;
    cvCvtColor(image, gtkMask, CV_BGR2RGB);
    pix = gdk_pixbuf_new_from_data((guchar *) gtkMask->imageData,
                       GDK_COLORSPACE_RGB,
                       FALSE,
                       gtkMask->depth,
                       gtkMask->width,
                       gtkMask->height,
                       gtkMask->widthStep, NULL, NULL);
    rpix = gdk_pixbuf_scale_simple(pix, dst_w, dst_h, GDK_INTERP_BILINEAR); //rescale image
    g_object_unref(pix);  //free the intermediate pixbuf...
    return rpix;
}

Ex. 3b)

Para passar um pixbuf para uma janela de DrawingArea, recomenda-se usar as funcionalidades cairo (típicos do GTK+3 mas também presentes nas últimas versões do GTK+2).

Assim, criar em myf.c a função pari_RefreshDrawingArea que aceita dois argumentos: uma string com o nome do widget que será "pintado" (uma drawing area) e a imagem Ipl do openCV. Assim, os passos principais são converter de openCV para GTK e depois usar as funcionalidades do cairo para desenhar na janela. Exempo de uma solução possível:

//copy from IplImage to pixbuf and paint DA
void pari_RefreshDrawingArea( char * widgetName, IplImage *img)
{
        GtkWidget *da=GTK_WIDGET(gtk_builder_get_object (builderG, widgetName));
        if( ! da )
        {
          printf("failed\n");
          return;
        }
        GdkPixbuf *pix=pari_ConvertOpenCv2Gtk(img, da->allocation.width, da->allocation.height );
        cairo_t *cr = gdk_cairo_create (gtk_widget_get_window(da));
        gdk_cairo_set_source_pixbuf(cr, pix, 0, 0);
        cairo_paint(cr);
        cairo_fill(cr);
        cairo_destroy(cr);
        g_object_unref(pix);  //free the pixbuf...
}

        pari_PerformImageAcquisition(captureG);             //acquire new image

Ex. 4b)

Criar a função void pari_ProcessUserOperations(IplImage *src, IplImage *dst) que fará as diversas operações sobre src e coloca o resultado em dst.

Esta função invocará as diferentes operações de processamento de imagem uma após outra e, expectavelmente, acumulando os efeitos das diversas operações. Assim, serão invocadas tantas funções quantas as opções dos botões das caixas assinaladas e que serão decididas pelo aluno.

Estas funções terão o protótipo e nome genérico:

/**
 * @brief  Function to perform some specific operation.
 *
 * @param  src Source image to process
 * @param  dst Image to put the processed result
 * @param  gb  GtkBuilder pointer for the currently loaded GUI.
 * @param  udata1 gpointer for generic data for the operation. Normally the widget name. Return if NULL.
 * @param  udata2 gpointer for generic data for the operation. Normally the widget name for auxiliary parameter. May be ignored.
 */
void pari_UserOperationN(IplImage *src, IplImage *dst, GtkBuilder *gb, gpointer udata1, gpointer udata2);

Onde N será 1,2,3,4, etc., conforme o número da operação a implementar.

Estrutura parcial de pari_ProcessUserOperations. Falta definir o último parâmetro de cada operação.
void pari_ProcessUserOperations(IplImage *src, IplImage *dst)
{
        /*If nothing is to be done at least the dst is the same as src*/
        cvCopy( src, dst, NULL);

        //Now operate all functions accumulated (one after one)
        pari_UserOperation1(dst, dst, builderG, (gpointer)"checkbutton1", ...);
        pari_UserOperation2(dst, dst, builderG, (gpointer)"checkbutton2", ...);
        pari_UserOperation3(dst, dst, builderG, (gpointer)"checkbutton3", ...);
        pari_UserOperation4(dst, dst, builderG, (gpointer)"checkbutton4", ...);
        pari_UserOperation5(dst, dst, builderG, (gpointer)"checkbutton5", ...);
        pari_UserOperation6(dst, dst, builderG, (gpointer)"checkbutton6", ...);
}

Exercício 5 - Criação cooperativa das funções pari_UserOperationN()

A partir deste exercício, o trabalho será feito em colaboração através do GitHub. Assim, devem ser constituídos grupos de dois alunos (eventualmente de três) que trabalharão em colaboração para desenvolver as funções de processamento que faltam.

A criação de grupos pode-se fazer por alunos sentados lado a lado para agilizar as conversas, mas não é forçoso que assim seja…

Ex. 5a)

Cada membro do grupo faz o git clone do repositório do respetivo colega.

Cada membro faz um pedido de colaboração através do envio de issues para desenvolver uma função do grupo das 6 que estarão associadas a cada um dos botões de "checkbox".

O pedido deve dizer qual o número do botão (Function1, Function2, etc.) e qual o significado do último parâmetro da função).

Exemplo de um pedido a comunicar por uma "issue":

Implementar na função 2 (UserOperation2) uma operação
de binarização (threshold) em que a ativação está na checkbox
cujo nome está no *udata1* e o limiar de binarização está numa
barra deslizante cujo nome está em *udata2*.

Ou seja, quem faz o pedido espera que o colaborador desenvolva uma função chamada pari_UserOperation2() que ele chamará com:

pari_UserOperation2(dst, dst, builderG, (gpointer)"checkbutton2",(gpointer)"adjustment1");

para fazer a binarização da imagem dst (primeiro argumento) e que a coloque na variável do segundo argumento (aqui de novo em dst para acumular efeitos) quando carregar no botão "checkbutton2" e/ou alterar a leitura da barra "adjustment1".

Ex. 5b)

Cada uma destas funções pari_UserOperationN() tem de verificar o estado do botão e outros widgets que o operador esteja a manobrar.

Formato das imagens

As imagens adquiridas são a cores, mas nem todas as operações são possíveis a cores. Nesse caso, a imagem original deve ser convertida numa nova imagem a níveis de cinzento, aplicar-lhe a operação, e depois replicar (fazer o cvMerge) de três cópias dessa imagem tratada para a imagem de destino (que será também de 3 camadas). Isto é necessário nestes exercícios para se manter sempre o mesmo formato e estrutura da imagem para facilitar o uso de código e das variáveis. Em rigor não é preciso replicar três imagens em níveis de cinzento para ter uma imagem em niveis de cinzento. Na verdade, as imagens que se visualizam são todas de 3 canais (a cores) mas quando os 3 canais são iguais há a ilusão de se ter uma imagem a níveis de cinzentos. Indicam-se de seguida exemplos ilustrativos.
Conversão para cinzento

A conversão para cinzento exige alguma atenção. Se a imagem tiver que ser convertida para cinzento, o mais fácil é criar uma imagem temporária, fazer a conversão, replicar os três canais iguais na imagem de destino e depois libertar a imagem temporária criada.

      IplImage *tmp_gray=cvCreateImage( cvSize(src->width,src->height), IPL_DEPTH_8U, 1);  //aux image
      cvCvtColor(src, tmp_gray, CV_RGB2GRAY);
      cvMerge(tmp_gray, tmp_gray, tmp_gray, NULL, dst); //thee equal channels
      cvReleaseImage(&tmp_gray);   //free aux image
Aplicação de Canny

A operação de deteção de arestas com Canny tem de ser feita numa imagem em cinzento. Assim, para aplicar um Canny será preciso converter a imagem a cores numa de cinzento, aplicar o Canny e, de novo, replicar os canais (todos iguais) para termos uma imagem a 3 canais.

      IplImage *tmp_gray=cvCreateImage( cvSize(dst->width,dst->height), IPL_DEPTH_8U, 1);  //aux image
      cvCvtColor(dst, tmp_gray, CV_RGB2GRAY);
      cvCanny(tmp_gray, tmp_gray,0,30,3);
      cvMerge(tmp_gray, tmp_gray, tmp_gray, NULL, dst);
      cvReleaseImage(&tmp_gray);                          //free aux image

Os limiares de Canny poderão ser outros!

Exercício 6 - Geração da documentação em cmake

Para gerar a documentação com Doxygen CMakeLists.txt, alguns passos standard são recomendados:

1-Criar um ficheiro de configuração genérico: Doxyfile.in  (exemplo on-line)
2-Criar um ficheiro de mainpage.dox para a descrição geral do projeto (exemplo on-line)
3-Criar um target dedicado e outras instruções na CMakeLists.txt
4-Criar eventuais ficheiros auxiliares à documentação (como um logotipo, etc.) (exemplo on-line)

Assim, a CMakeLists.txt leva instruções adicionais, como exemplificado a seguir:

#-------- Adições à CMakeLists.txt existente ----
# The project version number. Used in documentation generation and similar
set(VERSION_MAJOR   1   CACHE STRING "Project major version number.")
set(VERSION_MINOR   0   CACHE STRING "Project minor version number.")
set(VERSION_PATCH   0   CACHE STRING "Project patch version number.")
mark_as_advanced(VERSION_MAJOR VERSION_MINOR VERSION_PATCH)

#-- add a target to generate API documentation with Doxygen
find_package(Doxygen)
option(BUILD_DOCUMENTATION "Create and install doc (requires Doxygen)" ${DOXYGEN_FOUND})
if(BUILD_DOCUMENTATION)
    if(NOT DOXYGEN_FOUND)
        message(FATAL_ERROR "Doxygen is needed to build the documentation.")
    endif()
    set(doxyfile_in "${CMAKE_CURRENT_SOURCE_DIR}/doc/Doxyfile.in" )
    set(doxyfile "${CMAKE_CURRENT_BINARY_DIR}/Doxyfile")
    set(doxy_main_page "${CMAKE_CURRENT_SOURCE_DIR}/doc/mainpage.dox")
    configure_file(${doxyfile_in} ${doxyfile} @ONLY)
    add_custom_target(doc
        COMMAND ${DOXYGEN_EXECUTABLE}
        WORKING_DIRECTORY "${CMAKE_CURRENT_BINARY_DIR}"
        COMMENT "Generating API documentation with Doxygen"
        VERBATIM)
endif()
#code inspired on: http://www.tty1.net/blog/2014/cmake-doxygen_en.html
#---------------------------------------------------

Algumas variáveis são definidas na CMakeLists.txt para depois serem modificadas externamente (em especial na Doxyfile.in). A package Doxygen é procurada e se existir, várias configurações são feitas e cria-se o target doc, para que um "make doc" venha a gerar a documentação dentro da pasta build. Os ficheiros ligados à geração de documentação devem em geral ficar numa pasta à parte, como por exemplo doc e assim manter o projeto mais organizado.
